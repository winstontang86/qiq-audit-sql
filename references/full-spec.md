# 后台 DB 规范（完整版）

> 本文为「workbuddy 后台 DB 规范建议」的全文整理版本，按章节组织。每条规范都标注了等级（**必须** / **推荐** / **可选**）。本 skill 当前只检查「必须」与「推荐」两个等级，「可选」条目暂不做强制检查。

## 适用范围

- 主要面向**新建库表**。存量库表稳定起见尽量规范化，不方便调整的保持现状。
- 适用于代码（源码内嵌 SQL）与独立 `.sql` 文件。

---

## 1 建库建表规范（含表结构修改操作）

### 1.1 通用重点规范

- **1.1.1【必须】禁止使用保留字**。MySQL 将保留字用于特定语法解析与操作，库名/表名/字段名与关键字相同会导致 SQL 解析失败。
- **1.1.2【推荐】数据库存储引擎默认 InnoDB**；有事务的情况下必须选 InnoDB；其他选型需说明理由。
- **1.1.3【必须】建数据库时必须显式指定字符集**，字符集与字符序必须统一且唯一，且只能是 `utf8` 或 `utf8mb4`，**推荐 `utf8mb4`**。
- **1.1.4【推荐】建表语句字符集与库对齐**；存量库已建表的，尽量与原表对齐避免隐式转换导致索引失效。其他情况推荐 `utf8mb4`。
- **1.1.5【推荐】表名大小写不敏感**（纯小写 + 下划线，由 `lower_case_table_names` 控制）。
- **1.1.6【推荐】字段名排序规则不要使用大小写敏感的**，例如不要 `utf8_bin`，改用 `utf8_general_ci` / `utf8mb4_general_ci` / `utf8mb4_0900_ai_ci`。
- **1.1.7【必须】禁止 `float` / `double`**，小数必须使用 `decimal`；超出 decimal 范围则拆成整数与小数分开存储。
- **1.1.8【推荐】优先使用整数替代小数**。
- **1.1.9【必须】建表必须显式声明主键**，不能省略让 MySQL 用默认隐式主键。原因：影响主从同步（从库回放 binlog 全表扫描，导致延迟）；非自增/带业务属性主键会导致写入性能差。
- **1.1.10【必须】主键建议使用 `bigint unsigned`**。
- **1.1.11【必须】新建表的索引字段必须 `NOT NULL`**。
- **1.1.12【推荐】除自增键和大字段外，建议使用 `NOT NULL`**。原因：NULL 会增加应用复杂度、聚合函数（COUNT/MIN/SUM）忽略 NULL、NULL 与任何值比较结果为 NULL、占用额外空间。旧表新加字段需允许 NULL（避免长期持锁）。
- **1.1.13【推荐】非空约束的列推荐带默认值** `NOT NULL DEFAULT xx`。
- **1.1.14【推荐】`timestamp` / `datetime` 列建议添加默认值**。
- **1.1.15【必须】禁止使用 `ENUM` 类型**。扩展性差，可用 `tinyint`/`smallint`/`char` 替代。低版本或非严格模式下，往 enum 列插入无效值不会报错，会变成空字符串（数值为 0），存在隐患。
- **1.1.16【必须】索引禁止走极端**：每个列都建一个单列索引、或建一个覆盖所有列的全列索引，都不允许。
- **1.1.17【必须】禁止使用外键**。后续修复数据极其复杂。
- **1.1.18【推荐】表字段数控制在两位数以内（< 99）**。
- **1.1.19【推荐】表必备字段：主键 + 创建时间 + 更新时间**。主键叫 `id`（必为主键，类型 `bigint unsigned`，未来分库分表用雪花算法或类似生成且保持有序）；创建/更新时间字段建议 `datetime`，命名接受以下任一**命名族**：
  - 创建时间：`create_at` / `created_at` / `gmt_create` / `ctime`
  - 更新时间：`update_at` / `updated_at` / `gmt_modified` / `mtime`

  项目可通过根目录 `.audit-sql.json` 的 `column_aliases` 进一步扩展可接受的别名。
- **1.1.20【必须】所有字段、表必须带注释**，方便维护与交接。

### 1.2 命名相关

- **1.2.1【必须】表名/字段名只能用小写字母或数字**，禁止数字开头，禁止两个下划线之间只出现数字，不能以下划线结尾。MySQL 在 Windows 下不区分大小写但 Linux 下默认敏感；DB+Redis 组合时 Redis key 默认不区分大小写也会有兼容隐患。
- **1.2.2【推荐】索引命名**：主键索引 `pk_字段名`；唯一索引 `uk_字段名`；普通索引 `idx_字段名`。
- **1.2.3【推荐】表达「是否」概念的字段**用 `is_xxx` 命名，类型 `tinyint unsigned`（1 = 是，0 = 否）。
- **1.2.4【推荐】库名/表名/列名长度不要超过 32 个字符**。
- **1.2.5【推荐】时间字段命名后缀**：时间戳（毫秒精度）以 `_at` 结尾；日期（天精度）以 `_date` 结尾；月份以 `_month` 结尾。

### 1.3 类型和索引相关

- **1.3.1【推荐】字符串长度近似相等时使用 `char`**。
- **1.3.2【推荐】时间类型选择**：大部分场景用 `datetime`；跨时区场景用 `bigint`（时间戳）+ `datetime`，需要区分时区还要加时区字段。`datetime` 8 字节，存绝对值；`timestamp` 4 字节，跟随时区变化。
- **1.3.3【推荐】不建议 `TEXT` / `BLOB`**。必须用则单独建表，主键关联。否则会浪费磁盘和内存空间，淘汰热数据，影响性能。
- **1.3.4【推荐】业务上具有唯一特性的字段（包括组合）必须建唯一索引**。即使应用层校验完善，没有唯一索引必然产生脏数据。
- **1.3.5【推荐】varchar 字段建索引时必须指定索引长度**，区分度即可，不必全字段。可用 `count(distinct left(列名, 索引长度))/count(*)` 评估区分度，长度 20 通常区分度 > 90%。
- **1.3.6【可选】更多索引细节参阅公司 SQL 规范**：https://git.woa.com/standards/sql

## 2 库表操作规范

- **2.1【必须】禁止 SQL 命令直接删库**，必须走标准下线流程：下线前周知评估 → 下线前一个月流量观察 → 数据存档备份 → 数据库下线。
- **2.2【必须】禁止 SQL 命令直接删表**，走表下线流程：下线周知评估 → 下线前流量观察 → 存档备份 → 表下线。
- **2.3【推荐】对同一表的多条修改语句建议合并为一条 `ALTER`**。
- **2.4【必须】禁止删除列**。
- **2.5【必须】`INSERT` 必须显式指定字段名**。原因：表的字段数量和顺序可能被修改。
- **2.6【必须】`UPDATE` / `DELETE` 必须带 `WHERE` 条件；`SELECT` 也禁止不带 `WHERE`**（避免无谓全表扫描）。批量更新/删除一次行数 < 1w 行，循环执行；删整表用 `TRUNCATE`（注意 truncate 不触发 trigger）。
- **2.7【推荐】`WHERE` 等号左右字段类型必须一致**，避免隐式转换导致索引失效。
- **2.8【推荐】尽量不要用 `SELECT *`**，明确指定字段。原因：浪费网络、字段更新后 model 层未同步会报错。
- **2.9【推荐】索引列不要使用函数表达式**，否则索引失效。
- **2.10【必须】超过三个表禁止 JOIN**。被 JOIN 字段类型必须绝对一致，多表关联字段必须有索引。
- **2.11【必须】禁止使用存储过程**。难调试、难扩展、不可移植，大并发下有额外开销。
- **2.13【推荐】不要全模糊/左模糊**：`like '%abc%'` 或 `like '%abc'`，可以用 `like 'abc%'`。
- **2.14【推荐】`WHERE IN()` 元素数量控制在 200 以内**。
- **2.15【推荐】用 `ISNULL()` 判断 NULL**。`NULL = NULL` 返回 NULL 不是 true；`NULL <> NULL` 返回 NULL 不是 false；`NULL <> 1` 返回 NULL 不是 true。
- **2.16【推荐】用 `count(*)` 而不是 `count(列名)` / `count(常量)`**。`count(*)` 是 SQL92 标准统计行数语法，与 NULL 无关；`count(列名)` 不统计 NULL 行。
- **2.17【推荐】超大分页用延迟关联或子查询优化**。MySQL 不是跳过 offset 行，而是取 offset+N 行后丢弃前 offset 行。
- **2.18【推荐】禁止 `ORDER BY RAND()`**。会把所有符合条件的数据装载到内存做随机排序，消耗 CPU/IO/内存。

## 3 DB 选型和参数建议

- **3.1【推荐】新建数据库使用 MySQL 8.0**，不低于 5.7。
- **3.2【推荐】数据量大时考虑分库分表**：两三年后超过 2000 万行需要分。能力够用 TDSQL 分布式版本。
- **3.3【推荐】启用 `sql_mode` 严格模式**。
- **3.4【推荐】选型参考公司规范**。

---

## 4 DB 部署和监控规范

- **4.1【必须】所有 DB 必须有跨 AZ 主从节点**，重要数据必须实现跨 Region 同步。
- **4.2【必须】所有 DB 必须统一配置重要指标监控告警**：连接数、连接使用率、死锁数、磁盘利用率、内存利用率等。
- **4.3【推荐】DB 监控告警统一由运维管理**。
- **4.4【推荐】慢查询监控与统计**，每个一定周期（一周）review 与优化。

---

## 5 安全和其他

- **5.1【推荐】数据库敏感信息字段必须加密存储**。
- **5.2【推荐】对接 DB 的逻辑层负责 SQL 转义**，防止 SQL 注入。
- **5.3【必须】MySQL Client SDK 初始化必须显式配置连接超时时间**。C++ SDK 默认 0 表示采用系统设置（约 180s），主备切换时容易卡住，长连接尤甚。

---

## 6 生产数据库管理规范

- **6.1【必须】开启审计日志**。现网需要根据审计日志排查 CDB 异常（如不定时 CPU 上涨）。
- **6.2【必须】新增 RO 组时必须是两个不同可用区的 RO**，防单 AZ 故障。
- **6.3【必须】主实例必须开启延时 RO**，防异常数据引发内核 bug 导致 CDB 全局故障。

## 附录：豁免与项目级配置

### A.1 内联豁免注释

在 SQL 注释（`-- ...` 或 `/* ... */`）中使用以下指令可豁免指定规则。**只识别 SQL 注释，不识别宿主语言（如 Go `// ...`）的原生注释**。

```sql
-- audit-sql:disable-file=1.2.1,1.1.20  reason=upstream keycloak schema
/* audit-sql:disable-file=*  reason=test fixture */

-- audit-sql:disable-next-line=1.2.1  reason=月分表占位符
CREATE TABLE t_log_YYYYMM (...);
```

- `disable-file=...` ：豁免全文件。
- `disable-next-line=...` ：仅豁免该注释之下一行起始的 SQL 语句。
- 规则号可多个逗号分隔；`*` 表示豁免所有规则。
- `reason=...` 不是必填但强烈建议写上，便于 code review。

### A.2 项目级配置文件 `.audit-sql.json`

放在仓库根目录，脚本会从被检查路径向上递归查找。首版支持以下 3 个字段：

```json
{
  "column_aliases": {
    "create_at": ["created_at", "gmt_create"],
    "update_at": ["updated_at", "gmt_modified"]
  },
  "table_name_placeholders": ["_YYYYMM", "_YYYYMMDD"],
  "excluded_paths": ["**/keycloak-bridge/**", "**/test/fixture/**"]
}
```

- `column_aliases`：扩展 1.1.19 可接受的必备字段别名（在内置默认之上合并）。
- `table_name_placeholders`：表名中包含该片段时跳过 1.2.1 命名检查（适用于月/日分表占位符）。
- `excluded_paths`：**不会豁免**，但命中后该文件的违规在报告里会被划入「⚪ 上游/历史兼容」分组，避免息意上动不能动的代码型成阐断噪声。如果要真豁免，请配合 A.1 的 `disable-file`。

### A.3 CI / 合流门禁场景建议用法

```bash
# 在 PR 上只对本次变更的新增/修改行严格检查
python3 audit_sql.py . --diff origin/master...HEAD --strict
```

- 新代码上线需严格合规；历史代码逐步治理，避免被存量违规一次性阐断。
- 豁免与配置会同步生效于增量检查。
